项目背景
为什么需要 FileFinder？
Windows 自带搜索速度慢、功能单一，现有工具要么收费、要么臃肿，我们需要一个轻量、快速、功能完整的本地搜索方案。
🐌
Windows 搜索慢
系统自带搜索建立索引耗时，搜索响应经常数秒甚至数十秒，体验极差
💰
专业工具收费
Listary、Everything 等优秀工具核心功能需付费，且不支持内容搜索
📦
功能单一
多数工具只支持文件名搜索，无法搜索文件内容，缺少预览功能
▼
✅ FileFinder 的解决方案
基于 SQLite 持久化索引 + 内存缓存，一次扫描终生复用，文件名搜索毫秒级响应；
支持 4 种匹配模式（模糊/精确/通配符/正则）+ 文件内容搜索（多线程并发 + FTS5 全文索引）；
内置 10+ 种格式预览，搜索即可预览，无需打开外部程序。

技术难点
多模式搜索引擎
如何在一个引擎中统一支持四种匹配模式，并保证搜索结果的精准排序？
难点 1 四种匹配模式的统一调度
通过 SearchQuery.name_mode 字段区分匹配模式，在 NameSearcher 中统一调度：
先用 _extract_pre_filter_pattern() 从通配符/正则中提取字面量，通过 SQL LIKE 预过滤缩小范围，
再在 Python 层用 fnmatch / re.search 精确匹配，避免全表扫描。
难点 2 相关度评分算法
设计四级评分体系，精确匹配 > 前缀匹配 > 词边界匹配 > 子串匹配，
同时加入模式长度权重，更长的匹配关键词获得更高分数：
匹配类型分数示例
精确匹配100+搜索 main.py，文件名恰好为 main.py
前缀匹配80+搜索 main，文件名以 main 开头
词边界匹配60+搜索 main，匹配 my_main_file.py
子串匹配40+搜索 main，匹配 remain.py
技术难点
内容搜索与全文索引
如何实现高效的文件内容搜索？双策略：FTS5 全文索引 + 实时扫描回退
⚡ FTS5 索引搜索
  • 使用 SQLite FTS5 虚拟表构建全文索引
  • 扫描时 jieba 分词写入索引
  • 搜索时对查询文本分词后 MATCH
  • BM25 算法排序，毫秒级响应
  • 从 file_content_raw 提取上下文行
🔄 实时扫描回退
  • 无索引时自动回退到实时扫描
  • ThreadPoolExecutor 多线程并发
  • 4~8 个线程逐文件搜索
  • 字面量预过滤快速排除不匹配文件
  • 支持纯文本 + PDF 逐页搜索

UI 渲染条/批
数据库优化
SQLite WAL 模式（读写互不阻塞）+ 16MB 缓存 + 64MB 内存映射 + 4KB 页大小；
启动时全表加载到 SearchCache 内存缓存，文件名搜索零磁盘 IO；
FTS5 全文索引 + file_content_raw 原始内容存储，内容搜索毫秒级。
UI 性能优化
搜索防抖 300ms（Debouncer）避免频繁触发；
结果分批渲染（每批 50 条），初始显示 200 条，最多 1000 条；
所有耗时操作在 QThread / ThreadPoolExecutor 中执行，主线程不阻塞。
详细设计
文件解析器注册表
采用注册表模式（Registry Pattern），支持灵活扩展新的文件格式解析器
🏛️ 设计模式
  • FileParser 抽象基类定义接口
  • can_parse() 判断能否解析
  • parse() 提取文本内容
  • supported_extensions 声明支持的扩展名
  • ParserRegistry 按序遍历所有解析器
  • 新增格式只需实现子类并注册
📦 已实现的解析器
  • TextParser — 纯文本/代码（自动编码检测）
  • PDFParser — PDF（PyMuPDF 逐页提取）
  • DocxParser — Word（段落+表格提取）
  • XlsxParser — Excel（read_only 模式）
  • PptxParser — PPT（幻灯片文本+表格）
