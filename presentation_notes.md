## 第 1 页：封面

这一页就是项目名称和核心标签，没什么技术内容。标签的含义：

- **全文检索**：能在文件内容里搜索关键词

## 第 2 页：项目背景 — 为什么需要 FileFinder？

### 痛点 1：Windows 搜索慢

Windows 自带的搜索功能（资源管理器右上角的搜索框）速度很慢，因为它每次搜索都要实时遍历硬盘上的文件。想象一下，你的硬盘上有 10 万个文件，它要一个一个检查名字是否匹配，当然慢。

### 痛点 2：专业工具收费

- **Everything**：只支持文件名搜索，不支持搜索文件内容，高级功能收费
- **Listary**：核心功能收费，而且主要做快速启动，不是专门的搜索工具

### 痛点 3：功能单一

大部分搜索工具只能按文件名搜，不能搜文件内容。比如你想找"哪个配置文件里写了数据库密码"，它们做不到。

### FileFinder 的解决方案

核心思路是**先建索引，再搜索**：

1. **SQLite 持久化索引**：第一次使用时扫描硬盘，把所有文件信息（名字、路径、大小、修改时间等）存到 SQLite 数据库里。以后搜索直接查数据库，不用再遍历硬盘，所以很快。
2. **内存缓存**：程序启动时把整个数据库加载到内存里，搜索时直接在内存里查，连数据库都不用读，毫秒级响应。
3. **FTS5 全文索引**：不仅给文件名建索引，还给文件内容建索引。用 SQLite 的 FTS5（Full-Text Search 5）功能，搜索文件内容也是毫秒级。
4. **一次扫描，终生复用**：索引建好之后，通过文件系统监控（QFileSystemWatcher）自动检测文件变化，增量更新索引，不需要重新扫描。

## 第 3 页：功能概览 — 六大核心特性

### 🔍 多模式搜索

支持 4 种文件名匹配方式：

| 模式 | 说明 | 例子 |
|------|------|------|
| **模糊匹配** | 只要文件名里包含你输入的文字就算匹配 | 输入 `read` → 匹配 `README.md`、`spreadsheet.xlsx` |
| **精确匹配** | 文件名必须和你输入的完全一样 | 输入 `main.py` → 只匹配 `main.py`，不匹配 `mymain.py` |
| **通配符匹配** | 用 `*`（匹配任意多个字符）和 `?`（匹配一个字符） | 输入 `*.py` → 匹配所有 Python 文件 |
| **正则表达式** | 用正则表达式做高级模式匹配 | 输入 `\d+\.py` → 匹配 `123.py`、`1.py` |

### 📄 内容搜索

在文件内容里搜索关键词，不是搜文件名。比如搜索 `database`，会找到所有内容里包含 `database` 这个词的文件。

- **多线程并发**：同时用 4~8 个线程搜索不同文件，速度快
- **支持多种格式**：纯文本、代码、PDF、Word、Excel、PPT 都能搜

### ⚡ 极速索引

- **SQLite 持久化**：文件信息存在数据库里，关掉程序再打开还在
- **内存缓存**：启动时加载到内存，搜索不用读硬盘
- **FTS5 全文索引**：文件内容也建了索引，内容搜索也是毫秒级

### 👁️ 内容预览

选中搜索结果后，右侧面板直接显示文件内容，支持 10+ 种格式：

- 文本/代码（带语法高亮）
- PDF（分页渲染）
- 图片（支持缩放）
- Excel（表格展示，Sheet 切换）
- Markdown（渲染预览/源码切换）
- Word、HTML、压缩包文件列表、视频音频信息、文件夹内容

### 🔄 实时同步

用 Qt 框架提供的 `QFileSystemWatcher` 监控文件变化：

- 你在文件管理器里新建了文件 → 索引自动添加
- 你删除了文件 → 索引自动删除
- 你修改了文件 → 索引自动更新

不需要手动重新扫描。

### 🎯 精准筛选

- **9 大分类**：全部、文件夹、文档、代码、图片、视频、音频、压缩包、其他
- **子分类**：比如"代码"下面有 Python、JS/TS、Java 等子分类
- **4 种排序**：名称、相关度、修改时间、文件大小
- **联合搜索**：同时用文件名和内容条件搜索，取交集

## 第 4 页：分层架构

### 什么是分层架构？

把代码按照职责分成不同的层，每层只做自己的事情，层与层之间通过接口通信。好处是：修改某一层不会影响其他层，代码更清晰、更好维护。

### 五层结构详解

```
┌─────────────────────────────────────────────────┐
│  UI 层（用户界面层）                                │
│  负责显示界面和处理用户操作                            │
│  MainWindow, SearchBar, ResultList, PreviewPanel  │
├─────────────────────────────────────────────────┤
│  Core 层（核心业务逻辑层）                           │
│  负责搜索算法、文件解析等核心功能                      │
│  SearchEngine, NameSearcher, ContentSearcher      │
├─────────────────────────────────────────────────┤
│  Model 层（数据模型层）                             │
│  定义数据结构，不包含业务逻辑                          │
│  FileItem, SearchQuery, SearchResult              │
├─────────────────────────────────────────────────┤
│  Data 层（数据访问层）                              │
│  负责数据库读写操作                                  │
│  DatabaseManager, SearchCache, HistoryDAO          │
├─────────────────────────────────────────────────┤
│  Utils 层（工具函数层）                             │
│  提供通用工具函数，无状态                             │
│  encoding.py, path_helper.py, tokenizer.py        │
└─────────────────────────────────────────────────┘
```

**各层职责：**

| 层 | 职责 | 举例 |
|----|------|------|
| **UI 层** | 显示界面，接收用户输入，展示搜索结果 | 搜索栏输入文字 → 调用 Core 层搜索 → 把结果显示到列表 |
| **Core 层** | 实现搜索算法，不直接操作界面 | NameSearcher 从数据库查文件名，ContentSearcher 搜索文件内容 |
| **Model 层** | 定义数据长什么样，纯数据结构 | FileItem 描述一个文件有哪些属性（名字、大小、路径等） |
| **Data 层** | 和数据库打交道，读写数据 | DatabaseManager 执行 SQL 查询，SearchCache 在内存里缓存数据 |
| **Utils 层** | 提供通用工具函数 | encoding.py 检测文件编码，path_helper.py 处理路径 |

**关键规则：上层可以调用下层，下层不能调用上层。** 比如 UI 层可以调用 Core 层搜索，但 Core 层不能直接修改 UI。

## 第 5 页：核心类结构

### SearchEngine（搜索调度引擎）

**作用**：搜索的"总指挥"，协调文件名搜索和内容搜索。

**工作流程**：
1. 收到搜索请求（SearchQuery）
2. 如果有文件名条件 → 调用 NameSearcher 搜索文件名
3. 如果有内容条件 → 调用 ContentSearcher 搜索内容
4. 如果两个条件都有 → 分别搜索后取交集（联合搜索）
5. 按相关度排序，返回结果

**为什么需要它？** 因为搜索逻辑比较复杂（文件名搜索、内容搜索、联合搜索），需要一个统一的入口来调度，而不是让 UI 层直接调用各个搜索器。

### NameSearcher（文件名搜索器）

**作用**：根据文件名搜索文件，支持 4 种匹配模式。

**关键方法**：
- `fuzzy_match()`：模糊匹配，用 Python 的 `in` 操作符检查子串
- `wildcard_match()`：通配符匹配，用 Python 标准库 `fnmatch`
- `regex_match()`：正则匹配，用 Python 标准库 `re.search`
- `_calculate_relevance()`：计算相关度评分（下一页详细讲）

**优化策略 — 预过滤**：
如果用户用通配符搜 `test_*.py`，不会把数据库里所有文件都拿出来逐个匹配。而是先从 `test_*.py` 中提取出 `test_` 这个"字面量"（确定的文字部分），用 SQL `LIKE` 先过滤出名字包含 `test_` 的文件，再对这些文件做通配符匹配。这样大大减少了需要检查的文件数量。

### ContentSearcher（内容搜索器）

**作用**：在文件内容中搜索关键词。

**两种搜索策略**：
1. **FTS5 索引搜索**（优先）：如果有全文索引，直接查索引，毫秒级
2. **实时扫描**（回退）：没有索引时，多线程逐文件读取内容搜索

**Qt 信号（Signal）**：
- `progress_updated`：搜索进度更新（已处理 X 个文件 / 共 Y 个文件）
- `result_found`：找到一个匹配结果
- `search_completed`：搜索完成

**为什么用信号？** 因为内容搜索在后台线程执行，不能直接操作 UI。通过 Qt 的信号槽机制，后台线程发射信号，UI 线程接收信号并更新界面，这是线程安全的做法。

### DatabaseManager（数据库管理器）

**作用**：所有数据库操作的入口，单例模式。

**什么是单例模式？** 整个程序只有一个 DatabaseManager 实例。因为数据库连接需要统一管理，如果到处创建新的数据库管理器，会导致连接冲突和资源浪费。

**关键组成部分**：
- **SQLite 数据库**：存储文件索引和内容索引
- **SearchCache**：内存缓存，启动时加载全部索引数据

**数据库表结构**：
- `file_index_cache`：文件索引（路径、名字、扩展名、大小、修改时间等）
- `file_content_raw`：文件原始文本内容（用于提取上下文）
- `file_content_fts`：FTS5 全文索引虚拟表
- `search_history`：搜索历史
- `settings`：设置项

### ParserRegistry（文件解析器注册表）

**作用**：根据文件类型选择合适的解析器来提取文本内容。

**设计模式 — 注册表模式（Registry Pattern）**：
- 定义一个抽象基类 `FileParser`，规定所有解析器必须有 `can_parse()` 和 `parse()` 两个方法
- 每种文件格式实现一个解析器子类（TextParser、PDFParser、DocxParser 等）
- `ParserRegistry` 维护一个解析器列表，按顺序尝试每个解析器

**好处**：要支持新的文件格式，只需要写一个新的 Parser 子类并注册到列表里，不用修改其他代码。

### SearchQuery（搜索查询条件）

**作用**：把用户输入的搜索条件封装成一个数据对象。

**包含的字段**：
- `name_query`：文件名关键词
- `name_mode`：文件名匹配模式（fuzzy/exact/wildcard/regex）
- `content_query`：内容关键词
- `file_types`：文件类型过滤
- `include_dirs`：搜索范围目录
- `size_min`/`size_max`：文件大小范围

**为什么用数据类？** 如果不用 SearchQuery，搜索函数就需要十几个参数，调用起来很麻烦。用数据类把所有参数打包在一起，传递方便，代码更清晰。

## 第 6 页：技术难点① — 多模式搜索引擎

### 难点 1：四种匹配模式的统一调度

**问题**：4 种匹配方式的搜索逻辑完全不同，怎么在一个引擎里统一处理？

**解决方案**：
1. 用 `SearchQuery.name_mode` 字段记录用户选择的匹配模式
2. 在 `NameSearcher.search_by_name()` 里用 if-elif 判断模式，调用对应的匹配函数
3. **预过滤优化**：不管哪种模式，都先从搜索词中提取"字面量"（确定的文字部分），用 SQL `LIKE` 预过滤

**预过滤的具体过程**：

```
用户输入通配符: test_*.py
  ↓ 提取字面量
字面量: test_
  ↓ SQL LIKE 预过滤
SELECT * FROM file_index_cache WHERE name LIKE '%test_%'
  ↓ 得到候选文件（比如 50 个）
  ↓ 再用 fnmatch 精确匹配
最终匹配的文件（比如 12 个）
```

如果不做预过滤，就要把数据库里 10 万个文件全部拿出来逐个匹配，非常慢。预过滤后只需要检查 50 个候选文件，快得多。

### 难点 2：相关度评分算法

**问题**：搜索结果怎么排序？用户搜 `main`，`main.py` 应该排在 `remain.py` 前面。

**解决方案 — 四级评分体系**：

| 匹配类型 | 分数 | 含义 |
|----------|------|------|
| 精确匹配 | 100 + 模式长度 | 文件名和搜索词完全一样 |
| 前缀匹配 | 80 + 模式长度 | 文件名以搜索词开头 |
| 词边界匹配 | 60 + 模式长度 | 搜索词出现在分隔符（`_`、`-`、`.`、空格）后面 |
| 子串匹配 | 40 + 模式长度 | 搜索词出现在文件名中间 |

**为什么要加"模式长度"？** 因为更长的搜索词意味着更精确的匹配。比如搜 `main.py`（8 个字符）比搜 `m`（1 个字符）更精确，应该得到更高分数。

**词边界是什么？** 比如 `my_main_file.py`，`main` 前面有下划线 `_`，这个下划线就是"词边界"。说明 `main` 是一个独立的词，不是 `remain` 的一部分，所以分数更高。

## 第 7 页：技术难点② — 内容搜索与全文索引

### 双策略设计

**问题**：文件内容搜索很慢（要逐个打开文件读取内容），怎么优化？

**方案**：两种策略，自动选择：

#### 策略 1：FTS5 全文索引搜索（快）

**什么是 FTS5？** FTS5（Full-Text Search 5）是 SQLite 内置的全文搜索引擎。它的工作原理是：

1. **建索引时**：扫描文件内容 → 用 jieba 分词 → 把分词结果存入 FTS5 虚拟表
2. **搜索时**：对搜索词分词 → 用 `MATCH` 语法查询 FTS5 表 → 返回匹配的文件

**什么是 jieba 分词？** 中文没有天然的词间空格，需要算法来切分。jieba 是最流行的中文分词库，它把"数据库配置文件"切分成"数据库"/"配置"/"文件"三个词。

**什么是 BM25 排序？** BM25 是信息检索领域最经典的排序算法，它根据关键词在文档中出现的频率和文档长度来计算相关度。出现次数多、文档短的排名更高。

**FTS5 搜索流程**：
```
用户输入 "数据库配置"
  ↓ jieba 分词
"数据库" OR "配置"
  ↓ FTS5 MATCH 查询
找到包含这些词的文件
  ↓ 从 file_content_raw 获取原文
提取上下文行（匹配行前后各 3 行）
  ↓ 返回结果
```

#### 策略 2：实时扫描（慢但不需要索引）

**什么时候用？** 没有建立内容索引时自动回退到这种方式。

**工作流程**：
1. 收集搜索目录下所有可搜索的文件
2. 用 `ThreadPoolExecutor` 创建 4~8 个线程
3. 每个线程负责搜索一部分文件
4. 对每个文件：读取内容 → 检查是否包含关键词 → 如果匹配就记录结果

**优化措施**：
- **字面量预过滤**：先用简单的 `pattern in content` 检查，比正则快 5~10 倍
- **os.scandir**：比 `os.walk` 更快的目录遍历方式，利用系统缓存的文件信息
- **逐文件提交任务**：而不是按目录提交，负载更均衡
- **超时控制**：单文件 30 秒超时，总搜索 120 秒超时

### 为什么需要两种策略？

- FTS5 索引搜索快，但需要先花时间建索引（扫描所有文件内容并分词）
- 实时扫描慢，但不需要提前建索引
- 所以：有索引用索引（快），没索引就实时扫描（至少能用）

## 第 8 页：技术难点③ — 性能优化

### 数据库优化

#### WAL 模式（Write-Ahead Logging）

**默认模式的问题**：SQLite 默认用 DELETE 模式，写数据时要锁住整个数据库，读操作必须等写完才能继续。

**WAL 模式的好处**：写操作先写到 WAL 日志文件，不锁住数据库。读操作可以同时进行，读写互不阻塞。这是 SQLite 性能优化最关键的一步。

#### 其他 SQLite 优化

| 优化项 | 值 | 作用 |
|--------|-----|------|
| `cache_size` | 16MB | SQLite 内部缓存大小，越大查询越快 |
| `mmap_size` | 64MB | 内存映射大小，把数据库文件映射到内存，减少系统调用 |
| `page_size` | 4KB | 数据库页大小，和操作系统页大小对齐 |
| `synchronous` | NORMAL | 减少刷盘频率，提升写入速度（略微降低断电安全性） |
| `temp_store` | MEMORY | 临时表存在内存而非磁盘 |

#### SearchCache 内存缓存

**原理**：程序启动时，把 `file_index_cache` 表的所有数据加载到一个 Python 列表里。搜索时直接在列表里遍历查找，不需要执行 SQL 查询。

**为什么不用 SQL 查询？** 因为 Python 列表的遍历速度对于几万条数据来说已经足够快（毫秒级），而且避免了 SQLite 查询的开销（连接、编译 SQL、执行、返回结果）。

**线程安全**：用 `threading.Lock` 保证多线程访问缓存时不会出问题。

### UI 性能优化

#### 搜索防抖（Debouncer）

**问题**：用户每输入一个字符就触发搜索，输入 `main` 会触发 4 次搜索（`m`、`ma`、`mai`、`main`），浪费资源。

**解决方案**：防抖器。用户输入后等 300ms，如果 300ms 内没有新的输入才执行搜索。这样输入 `main` 只会触发 1 次搜索。

**实现方式**：用 Qt 的 `QTimer`，每次输入重置定时器，300ms 后定时器触发才执行搜索。

#### 分批渲染

**问题**：搜索返回 1000 个结果，一次性全部添加到列表里，界面会卡住。

**解决方案**：
1. 初始只显示前 200 条
2. 每批渲染 50 条，分批添加到列表
3. 列表底部显示"加载更多"按钮，每次加载 100 条

#### 后台线程

**问题**：搜索是耗时操作，如果在主线程（UI 线程）执行，界面会冻结，无法操作。

**解决方案**：
- 文件名搜索：在 `QThread` 中执行
- 内容搜索：在 `ThreadPoolExecutor` 中执行
- 搜索结果通过 Qt 信号槽传回主线程更新 UI

**为什么不能在子线程直接操作 UI？** Qt 的 UI 操作只能在主线程执行，否则会崩溃。信号槽是 Qt 提供的线程安全通信机制，子线程发射信号，主线程接收信号后更新 UI。

## 第 9 页：文件解析器注册表

### 什么是注册表模式？

**问题**：要支持 5 种文件格式的解析（文本、PDF、Word、Excel、PPT），以后可能还要加更多。如果用 if-elif 判断文件类型，每加一种格式就要改一堆代码。

**解决方案 — 注册表模式**：

1. 定义抽象基类 `FileParser`，规定所有解析器必须实现：
   - `can_parse(file_path)` → 能不能解析这个文件
   - `parse(file_path)` → 解析文件，返回文本内容
   - `supported_extensions` → 支持的文件扩展名

2. 每种格式实现一个子类：
   - `TextParser`：用 `charset_normalizer` 检测编码，读取文本
   - `PDFParser`：用 PyMuPDF（fitz）逐页提取文字
   - `DocxParser`：用 python-docx 提取段落和表格
   - `XlsxParser`：用 openpyxl 以 read_only 模式读取
   - `PptxParser`：用 python-pptx 提取幻灯片文字

3. `ParserRegistry` 维护一个解析器列表，按顺序尝试：
   ```python
   class ParserRegistry:
       def __init__(self):
           self._parsers = [TextParser(), PDFParser(), DocxParser(), XlsxParser(), PptxParser()]

       def parse(self, file_path):
           for parser in self._parsers:
               if parser.can_parse(file_path):
                   return parser.parse(file_path)
           return None
   ```

**好处**：要支持新格式（比如 .epub），只需要写一个 `EpubParser` 子类，加到列表里，其他代码完全不用改。

### 编码检测

**问题**：文本文件可能是 UTF-8、GBK、GB2312 等各种编码，用错误的编码打开会乱码。

**解决方案**（`encoding.py`）：
1. 先检查 BOM（文件开头的编码标记）：UTF-16 LE、UTF-16 BE、UTF-8 with BOM
2. 如果没有 BOM，用 `charset_normalizer` 库自动检测编码
3. 都检测不出来就默认 UTF-8
4. 读取时加 `errors='replace'`，遇到无法解码的字符用 `?` 替代，不会崩溃

## 附录：常见问题准备

验收时老师可能会问的问题和参考回答：

### Q1：为什么选择 PySide6 而不是 PyQt6？

PySide6 是 Qt 官方出品的 Python 绑定，LGPL 协议，可以免费用于商业项目。PyQt6 是第三方公司 Riverbank Computing 维护的，GPL 协议，商业使用需要付费。两者 API 几乎一样，但导入方式不同（PySide6 用 `Signal`，PyQt6 用 `pyqtSignal`）。

### Q2：SQLite 能处理多大数据量？

SQLite 单个数据库最大 281 TB，实际使用中百万级记录毫无压力。我们的场景（几十万文件索引）完全够用。关键是用了 WAL 模式和内存缓存来优化性能。

### Q3：FTS5 全文索引和实时扫描的区别？

| 对比 | FTS5 索引搜索 | 实时扫描 |
|------|-------------|---------|
| 速度 | 毫秒级 | 秒级（取决于文件数量） |
| 前提 | 需要先建索引 | 不需要 |
| 资源占用 | 占用更多磁盘空间 | 不占用额外空间 |
| 实时性 | 索引更新有延迟 | 搜索时实时读取 |

### Q4：联合搜索是怎么实现的？

1. 先搜索文件名，得到文件名匹配的文件集合 A
2. 再搜索文件内容，得到内容匹配的文件集合 B
3. 取 A 和 B 的交集（两个条件都满足的文件）
4. 按综合分数排序返回

### Q5：如何保证搜索不卡界面？

所有耗时操作都在后台线程执行：
- 磁盘扫描：`QThread`（ScanWorker）
- 文件名搜索：通过 `SearchCache` 内存缓存，本身就很快
- 内容搜索：`ThreadPoolExecutor` 多线程

搜索结果通过 Qt 信号槽传回主线程更新 UI，不会阻塞界面。

### Q6：文件编码检测是怎么做的？

1. 先检查 BOM 标记（UTF-16 LE/BE、UTF-8 with BOM 的文件开头有特殊标记）
2. 没有 BOM 就用 `charset_normalizer` 库统计分析文件内容的字节分布，推断最可能的编码
3. 读取时加 `errors='replace'`，无法解码的字符用 `?` 替代，保证不会崩溃

### Q7：相关度评分算法是怎么设计的？

四级评分：精确匹配(100) > 前缀匹配(80) > 词边界匹配(60) > 子串匹配(40)，再加上搜索词长度作为权重。这样更精确、更具体的匹配会排在前面。

### Q8：项目的数据流是怎样的？

```
用户输入搜索词
  → SearchBar 收集条件，构建 SearchQuery 对象
  → SearchEngine.search(query) 调度搜索
  → NameSearcher 查询 SearchCache / SQLite
  → ContentSearcher 搜索文件内容（FTS5 或实时扫描）
  → 合并结果，按评分排序
  → 通过信号槽传回主线程
  → ResultList 显示搜索结果
  → PreviewPanel 预览选中文件
```
